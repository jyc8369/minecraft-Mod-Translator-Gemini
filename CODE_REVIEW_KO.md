# 코드 리뷰 보고서

## 리뷰 범위

- 대상: `main.py`, `modules/` 하위 핵심 모듈, 빌드/실행 문서
- 관점: 안정성, 보안, 유지보수성, 사용자 경험, 테스트 가능성
- 결론: 데스크톱 GUI와 번역 파이프라인의 큰 흐름은 명확하지만, 압축 해제 보안과 중단 처리, GUI 스레드 안정성, 테스트 자동화 측면에서 개선 여지가 있습니다.

## 요약

| 우선순위 | 항목 | 영향 |
| --- | --- | --- |
| 높음 | `ZipFile.extractall()` 직접 사용 | 악성 JAR/ZIP의 경로 순회 항목이 임시 폴더 밖으로 쓰일 수 있습니다. |
| 높음 | 번역 중단 버튼의 실질적 중단 범위 제한 | Gemini 요청이나 청크 처리 중에는 즉시 중단되지 않아 사용자 기대와 다를 수 있습니다. |
| 중간 | GUI 객체를 백그라운드 스레드에서 직접 참조 | 일부 상태 조회가 Tk 메인 스레드 밖에서 일어나 플랫폼별 불안정성을 만들 수 있습니다. |
| 중간 | 테스트 코드 부재 | 압축/검색/JSON 파싱/청킹 회귀를 자동으로 검증하기 어렵습니다. |
| 낮음 | UI 언어 변경 흐름의 복잡성 | `_apply_ui_language()`가 존재하지만 실제 변경 흐름은 재시작에 의존합니다. |

## 주요 발견 사항

### 1. 압축 해제 보안 검증이 필요합니다

`modules/unzip_jar.py`는 JAR 파일을 `zipfile.ZipFile(...).extractall(...)`로 바로 해제합니다. ZIP 내부 항목에 `../` 또는 절대 경로가 포함된 경우, Python 버전과 환경에 따라 의도하지 않은 위치에 파일을 쓰는 Zip Slip 유형 문제가 발생할 수 있습니다.

**권장 조치**

- 각 `ZipInfo.filename`을 정규화한 뒤 최종 경로가 `extract_dir` 내부인지 확인합니다.
- 디렉터리 항목과 파일 항목을 분리 처리합니다.
- 비정상 경로가 발견되면 명확한 예외를 발생시키고 GUI에는 사용자 친화 메시지를 표시합니다.

### 2. 작업 중단 UX를 더 명확히 해야 합니다

`main.py`의 중단 이벤트는 JAR 단위 루프와 일부 단계 사이에서만 확인됩니다. Gemini API 호출이나 큰 청크 번역 중에는 사용자가 정지 버튼을 눌러도 요청이 끝날 때까지 대기할 수 있습니다.

**권장 조치**

- 청크 번역 루프에 취소 콜백 또는 `threading.Event`를 전달합니다.
- 각 청크 시작 전과 재시도 대기 전후에 중단 여부를 확인합니다.
- 즉시 취소가 불가능한 외부 API 호출 구간은 UI에 “현재 요청 완료 후 중단” 같은 상태 메시지를 표시합니다.

### 3. GUI 스레드 접근 경계를 정리하면 안정성이 좋아집니다

진행률 업데이트는 `after(...)`를 통해 메인 스레드로 위임하고 있어 좋은 구조입니다. 다만 컨트롤러의 작업 스레드에서 `self._app._input_lang_var.get()`, `self._app._output_lang_var.get()`, `self._app._t(...)`처럼 GUI 상태를 직접 읽는 부분이 있습니다.

**권장 조치**

- 번역 시작 시점에 입력 언어, 출력 언어, UI 문자열 또는 필요한 메시지를 메인 스레드에서 스냅샷으로 수집해 작업 스레드 인자로 넘깁니다.
- 컨트롤러가 `MainWindow`의 private 속성에 직접 접근하지 않도록 공개 메서드를 둡니다.

### 4. 번역 결과 검증은 좋지만 플레이스홀더 검증이 추가되면 안전합니다

`modules/gemini_translator.py`는 누락 key와 높은 동일값 비율을 감지해 재시도하는 방어 로직이 있습니다. 다만 Minecraft 번역 문자열에서 중요한 `%s`, `%1$s`, `%d`, `{0}` 같은 플레이스홀더가 번역 중 삭제되거나 변경되는지는 코드 레벨에서 검증하지 않습니다.

**권장 조치**

- 원문 값과 번역 값에서 플레이스홀더 토큰을 추출해 동일한 토큰 집합인지 비교합니다.
- 불일치 시 해당 key를 원문으로 되돌리거나 청크 재시도를 수행합니다.
- 경고 로그에 key와 누락 토큰을 남깁니다.

### 5. 자동 테스트 기반을 추가하는 것이 좋습니다

현재 저장소에는 별도 테스트 디렉터리나 테스트 러너 설정이 확인되지 않습니다. 압축, 언어 파일 탐색, JSON 응답 파싱, 청킹은 외부 API 없이도 단위 테스트가 가능합니다.

**권장 조치**

- `pytest` 기반 테스트를 추가합니다.
- 우선순위가 높은 테스트 후보:
  - `find_json()`이 `lang/*.json`을 우선순위대로 반환하는지
  - `_parse_json_payload()`가 코드펜스/앞뒤 설명이 섞인 JSON을 복구하는지
  - `_chunk_mapping()`이 key를 누락하지 않는지
  - `zip_jar()`/안전한 `unzip_jar()`가 경로와 파일 내용을 보존하는지

## 좋은 점

- OS별 설정 경로가 분리되어 있어 사용자 환경에 설정 파일을 저장하는 방식이 적절합니다.
- 번역 대상 JSON을 청크로 나누고, API 혼잡/일시 실패를 재시도하는 구조가 있습니다.
- 로그 창, 진행률, 결과 다이얼로그가 있어 장시간 작업의 상태를 사용자가 추적할 수 있습니다.
- UI 문자열이 `modules/i18n.py`에 모여 있어 한국어/영어 확장이 비교적 쉽습니다.

## 제안 작업 순서

1. `unzip_jar()`에 안전한 ZIP 항목 검증을 추가합니다.
2. `translate_json()`에 취소 콜백과 플레이스홀더 검증을 추가합니다.
3. GUI 상태 스냅샷을 작업 스레드 인자로 넘기도록 컨트롤러 경계를 정리합니다.
4. `pytest` 테스트 디렉터리와 최소 회귀 테스트를 추가합니다.
5. UI 언어 변경 정책을 “즉시 반영” 또는 “재시작 필요” 중 하나로 명확히 정리합니다.

## 검증 결과

- `python -m compileall main.py modules`로 현재 Python 소스의 문법 오류 여부를 확인했습니다.
